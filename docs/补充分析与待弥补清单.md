# HotFeed 补充分析与待弥补清单

> 基于 PRD v1.0 的深度追问分析

---

## 一、还需要弥补的关键问题

### 🔴 高优先级

#### 1. 去重算法的具体策略需要细化

当前只说"相似度 > 70%"，但实际场景更复杂：

| 场景 | 示例 | 挑战 |
|------|------|------|
| **同义表达** | "A明星离婚" vs "A明星官宣分手" | 分词后核心词相同 |
| **包含关系** | "某省发生地震" vs "某省发生6.5级地震致3人受伤" | 短标题被长标题包含 |
| **跨语言** | "China GDP growth" vs "中国GDP增长" | 需要翻译/跨语言对齐 |
| **时间演变** | 同一事件持续发酵，标题不断变化 | 需要事件追踪 |

**建议方案**：
- 短期：中文 jieba 分词 + TF-IDF 向量 + 余弦相似度（阈值 0.75）
- 中期：引入 NLP 模型（如 text2vec）做语义级去重
- 长期：事件追踪 + 实体识别

#### 2. 热度值的标准化问题

不同平台的热度值含义完全不同：
- 微博：搜索量（万）
- 抖音：播放量
- 百度：搜索指数
- 知乎：讨论量

**直接对比毫无意义**，需要标准化。建议：
- 方案 A：不看绝对值，只看排名（同平台内比较）
- 方案 B：Min-Max 归一化到 0-100 分
- 方案 C：使用排名百分位（Rank Percentile）

**推荐方案 B**：每条热点的"统一热度分" = (原始热度 - 该平台最小热度) / (该平台最大热度 - 该平台最小热度) × 100

#### 3. 国际平台数据获取的真实可行性

需要逐一验证：

| 平台 | 获取方式 | 可行性 | 备注 |
|------|---------|--------|------|
| Twitter/X | API v2 | ⚠️ 需付费（Basic $100/月） | 或用 Nitter RSS 替代 |
| Reddit | 公开 API | ✅ 免费 | `.json` 后缀即可 |
| Google News | RSS | ✅ 免费 | `news.google.com/rss` |
| BBC | RSS | ✅ 免费 | `feeds.bbci.co.uk/news/rss.xml` |
| CNN | RSS | ✅ 免费 | `edition.cnn.com/services/rss/` |
| YouTube | API v3 | ⚠️ 有配额限制 | 每天 10,000 单位 |

**风险提示**：Twitter 是国际热点核心来源，如果不用 API，Nitter RSS 可能不稳定。建议 MVP 阶段先用 Reddit + Google News + BBC/CNN 覆盖国际。

### 🟡 中优先级

#### 4. 缓存策略与数据一致性

10 分钟刷新一次，但各平台 API 本身有自己的更新频率：
- UAPIS：约 5 分钟更新
- RSS：各平台不同（10-60 分钟）

可能出现"看起来在更新，但数据没变"的情况。需要：
- 前端显示"数据更新时间"而非"页面刷新时间"
- 记录每条数据的 `fetched_at` 和 `data_version`
- 如果新数据和上次完全相同，不触发前端重渲染

#### 5. "点击本地查看"的降级方案

用户原始需求是"尽量本地打开"，但版权限制使得这很困难。建议的降级路径：

```
优先：新标签页打开原文（当前方案）
    │
    ├─ 如原文可嵌入（允许 iframe）
    │   └─ 在页面内 iframe 预览（类似 Google News）
    │
    └─ 如原文被墙/无法访问
        └─ 展示摘要 + 建议搜索关键词
```

#### 6. 移动端 PWA 化

既然是"网页 APP"，建议支持 PWA：
- 添加到主屏幕
- 离线缓存（上次数据）
- 推送通知（P3 功能的前置条件）

技术成本很低（manifest.json + Service Worker），但体验提升显著。

---

### 🟢 低优先级 / 远期

#### 7. SEO 与可发现性

作为公开网页，需要考虑：
- 服务端渲染 (SSR) 或预渲染，确保搜索引擎可索引
- 热点详情页（SEO 友好 URL）
- Open Graph 标签（分享到社交媒体时显示预览）

#### 8. 反爬与流量控制

如果产品受欢迎，可能面临：
- 其他站点爬取你的聚合数据
- API 调用量超出免费额度
- 需要 Rate Limiting + API Key 轮转策略

#### 9. 多语言支持

国际用户可能需要英文界面，建议前端框架支持 i18n，即使 MVP 只有中文。

## 三、MVP 的最小可行验证指标

在投入完整开发之前，建议先验证：

1. **技术可行性验证（1天）**
   - 调用 UAPIS API 获取微博+抖音+百度+知乎数据
   - 验证 RSS 获取 BBC + Google News 数据
   - 跑通去重 demo

2. **需求验证**
   - 找 5 个目标用户试用竞品（AnyKnew/今日热榜），问他们最不满意的点
   - 验证"国际+去重"是否真的是差异化价值
