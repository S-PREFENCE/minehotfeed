# HotFeed 热点聚合平台 - 产品需求文档 (PRD)

> **版本**：v1.1 | **日期**：2025-07-02 | **作者**：产品团队 | **状态**：已确认，进入研发评审阶段

---

## 一、产品概述

### 1.1 产品背景

在信息爆炸的时代，用户每天需要打开微博、抖音、知乎、百度、今日头条等多个 APP 才能全面了解当日热点，这导致：

- **时间浪费**：每次切换 APP 消耗注意力，单次切换成本约 10-15 秒，每日累计可达 15-30 分钟
- **信息遗漏**：各平台信息孤岛，用户容易错过跨平台的重要热点
- **重复阅读**：同一热点在多平台重复出现，信息冗余严重

### 1.2 产品定位

**HotFeed** 是一个一站式的全球热点聚合 Web APP，聚合国内 10+ 主流平台和国际 5+ 主流平台的热门头条，通过智能去重算法，让用户在 5 分钟内掌握当日全球热点。

### 1.3 目标用户

| 用户类型 | 占比估计 | 核心需求 | 使用频率 |
|----------|---------|---------|---------|
| 普通网民（信息消费者） | 50% | 快速了解今日热点 | 每日 1-2 次 |
| 媒体/内容从业者 | 20% | 选题参考、竞品监测 | 每日 3-5 次 |
| 投资者/分析师 | 15% | 舆情动态、市场情绪 | 盘中高频 |
| 海外华人/跨境人士 | 10% | 同时关注国内外动态 | 每日 2-3 次 |
| 学生/研究者 | 5% | 热点事件追踪 | 按需使用 |

### 1.4 核心价值主张

> **"一个页面，五分钟，全球热点尽在掌握"**

- **省时**：无需打开多个 APP，一站聚合
- **全面**：覆盖国内外 15+ 主流平台
- **去重**：同一热点只展示一次，信息密度更高
- **便捷**：Web APP 即开即用，无需安装

## 二、竞品分析

### 2.1 现有竞品对比

| 维度 | **AnyKnew** | **今日热榜 (Tophub)** | **鱼塘热榜** | **UAPIS 热榜** | **HotFeed（我们）** |
|------|------------|---------------------|-------------|---------------|-------------------|
| 覆盖平台 | 约 15 个 | 约 30 个 | 约 20 个 | 约 40 个 | 约 18 个（精选） |
| 国内/国际 | 仅国内 | 仅国内 | 仅国内 | 仅国内 | **国内 + 国际** |
| 去重能力 | ❌ 无 | ❌ 无 | ❌ 无 | ❌ 无 | ✅ **智能去重** |
| 分类筛选 | ❌ 无 | 按平台 | ❌ 无 | 按平台 | ✅ **按分类+平台** |
| 暗色模式 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 搜索功能 | ❌ | ❌ | ❌ | ✅（时光机） | ✅ |
| 响应式 | 部分 | 部分 | PC | PC | ✅ **全响应式** |
| 技术门槛 | 低 | 低 | 低 | 需开发 | 低（即开即用） |
| 数据更新 | 实时 | 实时 | 实时 | 5分钟 | **10分钟** |

### 2.2 竞品核心不足（我们的机会点）

1. **缺乏国际视角**：所有国内竞品都只覆盖国内平台，无法满足同时关注海外的用户
2. **无去重能力**：同一热点在微博、抖音、百度等多个平台重复展示，信息冗余
3. **分类缺失**：无法按科技/娱乐/财经等主题筛选
4. **移动端体验差**：多数竞品为 PC 优先设计，移动端体验粗糙
5. **无个性化能力**：所有用户看到相同内容，无法定制

## 四、功能需求（EARS 原则）

### 4.1 MVP (P0) - 核心功能

#### F-01 双区域头条展示

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Ubiquitous |
| **需求描述** | The system shall display two main sections on the homepage: "国内头条" (Domestic) and "国际头条" (International), each containing up to 50 unique hot topics. |
| **详细说明** | 页面左右双栏布局（PC）/ 上下布局（移动端），每区域最多展示 50 条热点，按综合热度排序 |
| **验收标准** | 1. 页面加载后 3 秒内展示两个区域 2. 每个区域不超过 50 条 3. 区域标题明确标注"国内头条"和"国际头条" |

#### F-02 智能去重

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Event-driven |
| **需求描述** | When multiple platforms report the same event, the system shall merge them into a single entry, displaying the highest-heat platform as the primary source and listing other platforms as secondary sources. |
| **去重策略** | 1. 标题相似度 > 70%（基于 TF-IDF 或中文分词相似度）视为同一事件 2. 取热度最高的平台作为主来源 3. 展示时标注"微博、抖音等 N 个平台热搜" |
| **验收标准** | 1. 同一事件在不同平台不重复展示 2. 合并后的条目标注多来源 3. 去重准确率 > 85% |

#### F-03 热点点击跳转

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Event-driven |
| **需求描述** | When a user clicks on a hot topic card, the system shall open the original source URL in a new browser tab. |
| **验收标准** | 1. 点击在新标签页打开原文 2. 不阻塞当前页面浏览 3. 支持浏览器默认行为（Ctrl+点击等） |

#### F-04 数据自动刷新

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Ubiquitous |
| **需求描述** | The system shall refresh hot topic data every 10 minutes automatically from all configured platforms. |
| **异常处理** | If a platform API fails, then the system shall retain the last successful data for that platform and display a "数据更新中" indicator. |
| **验收标准** | 1. 每 10 分钟自动刷新 2. 页面显示"上次更新：X分钟前" 3. 某平台失败不影响其他平台展示 |

#### F-05 响应式布局

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Ubiquitous |
| **需求描述** | The system shall adapt its layout responsively across desktop (≥1024px), tablet (768-1023px), and mobile (<768px) viewports. |
| **验收标准** | 1. PC 端双栏并排 2. 平板端双栏并排（缩小） 3. 手机端上下堆叠，国内在上国际在下 |

### 4.2 P1 - 重要功能

#### F-06 分类筛选

| 属性 | 内容 |
|------|------|
| **EARS 类型** | State-driven |
| **需求描述** | While the user is viewing hot topics, the system shall provide category filters (科技、娱乐、财经、体育、社会、国际) that filter displayed items without page reload. |
| **分类体系** | 科技 / 娱乐 / 财经 / 体育 / 社会 / 国际 / 全部 |
| **验收标准** | 1. 点击分类标签即时筛选 2. 选中标签高亮 3. 筛选后两个区域同步更新 |

#### F-07 关键词搜索

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Event-driven |
| **需求描述** | When a user enters a keyword in the search bar, the system shall filter and display matching hot topics from both domestic and international sections in real time. |
| **验收标准** | 1. 输入关键词后 500ms 内展示结果 2. 支持模糊匹配 3. 无结果时展示"未找到相关热点" |

#### F-08 平台来源筛选

| 属性 | 内容 |
|------|------|
| **EARS 类型** | State-driven |
| **需求描述** | While browsing, the system shall allow users to filter hot topics by source platform (e.g., show only Weibo + Douyin). |
| **验收标准** | 1. 可多选平台 2. 至少选择一个平台 3. 筛选即时生效 |

#### F-09 暗色模式

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Optional |
| **需求描述** | Where the user prefers dark mode, the system shall provide a toggle switch that applies a dark color theme across all pages. |
| **验收标准** | 1. 一键切换暗色/亮色 2. 偏好持久化（localStorage） 3. 所有组件适配暗色 |

### 4.3 P2 - 增强功能

#### F-10 热度趋势

| 属性 | 内容 |
|------|------|
| **EARS 类型** | State-driven |
| **需求描述** | While displaying each hot topic, the system shall show a trend indicator (rising 🔥 / stable ➡ / falling 📉) based on rank changes compared to the previous refresh cycle. |

#### F-11 用户偏好设置

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Optional |
| **需求描述** | Where the user has customized preferences, the system shall remember their preferred platforms and categories across sessions via browser localStorage. |

#### F-12 数据导出

| 属性 | 内容 |
|------|------|
| **EARS 类型** | Event-driven |
| **需求描述** | When a user clicks "导出", the system shall generate and download a CSV/JSON file of current hot topics. |

### 4.4 P3 - 远期规划

| 功能 | 说明 |
|------|------|
| F-13 浏览器推送通知 | 重大热点（热度突增 >200%）时浏览器推送通知 |
| F-14 热点时间线 | 查看某个热点事件在各平台的热度变化时间线 |

## 五、技术方案

### 5.1 架构选型：前后端分离（方案 B）

```
┌─────────────────────────────────────────────────────┐
│                    CDN / Nginx                        │
│              (静态资源 + 反向代理)                      │
└──────────┬──────────────────────────┬────────────────┘
           │                          │
     ┌─────▼─────┐             ┌──────▼──────┐
     │  前端 SPA  │             │  后端 API    │
     │  (静态部署) │◄───────────│  (定时抓取)  │
     │           │  REST API   │              │
     └───────────┘             └──────┬───────┘
                                      │
                               ┌──────▼───────┐
                               │   数据库       │
                               │  (SQLite/     │
                               │   PostgreSQL) │
                               └──────────────┘
```

### 5.2 推荐技术栈

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| **前端框架** | Vue 3 + Vite | 轻量、响应式、生态丰富 |
| **UI 框架** | Tailwind CSS | 快速开发、暗色模式内置 |
| **后端框架** | Python FastAPI | 异步支持好、适合 API 聚合场景 |
| **数据库** | SQLite（初期）/ PostgreSQL（扩展） | 轻量部署、零配置 |
| **定时任务** | APScheduler (Python) | 轻量级 Python 定时任务 |
| **部署** | Vercel (前端) + Railway/Render (后端) | 免费额度足够 MVP 使用 |
| **去重引擎** | jieba 分词 + difflib 相似度 | 中文分词 + 文本相似度计算 |

### 5.3 数据获取方案

**首选方案：UAPIS 免费热榜 API**
- 覆盖 40+ 平台，免费调用
- 统一的接口格式，减少开发量
- 接口地址：`https://uapis.cn/api/v1/misc/hotboard?type={platform}`

**备选方案：榜眼数据 (TophubData)**
- 覆盖平台更全面
- 支持历史数据和快照
- 需要 API Key，部分接口收费

**兜底方案：直接 RSS/爬虫**
- 对于 API 不覆盖的平台（如 Twitter），使用公开 RSS 或 HTML 解析
- 注意合规性和反爬策略

### 5.4 数据流

```
定时任务 (每10分钟)
    │
    ├──► UAPIS API ──► 国内平台数据 ──┐
    │                                  ├──► 去重引擎 ──► 数据库 ──► 前端 API
    └──► RSS/爬虫 ──► 国际平台数据 ──┘
```

## 八、验收标准

### 8.1 功能验收

- [ ] 国内区域展示 12 个平台的热点，总数不超过 50 条
- [ ] 国际区域展示 6 个平台的热点，总数不超过 50 条
- [ ] 同一事件在全局只出现一次（去重生效）
- [ ] 去重条目明确标注所有来源平台
- [ ] 点击热点在新标签页打开原始链接
- [ ] 每 10 分钟自动刷新数据
- [ ] 分类筛选即时生效
- [ ] 搜索功能返回匹配结果
- [ ] 暗色模式切换正常
- [ ] 页面在 PC / 平板 / 手机均正常显示

### 8.2 性能验收

- [ ] 首屏加载 (FCP) < 1.5 秒
- [ ] 最大内容绘制 (LCP) < 2.5 秒
- [ ] 累积布局偏移 (CLS) < 0.1
- [ ] API 响应时间 (P95) < 500ms

### 8.3 可靠性验收

- [ ] 单平台数据源故障不影响整体展示
- [ ] 全部数据源故障时展示上次缓存数据
- [ ] 后端服务异常时前端展示友好错误页
- [ ] 连续运行 7 天无内存泄漏

## 十一、附录

### A. 术语表

| 术语 | 说明 |
|------|------|
| 热点 (Hot Topic) | 在某个平台热榜上的条目 |
| 去重 (Deduplication) | 将多个平台报道的同一事件合并为一条 |
| 主来源 (Primary Source) | 合并条目中热度最高的平台 |
| 次来源 (Secondary Sources) | 合并条目中的其他平台 |
| 热度值 (Heat Score) | 平台原始热度数值（播放量/搜索量/讨论量） |
